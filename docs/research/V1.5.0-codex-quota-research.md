# V1.5.0 Codex 额度查询可行性调研

**调研时间**：2026-04-19
**调研目的**：判断多账户切换器能否在 UI 上展示每个 Codex OAuth 账户的"今天用了多少 / 还剩多少"

---

## 结论速览

| 问题 | 结论 |
|---|---|
| 1. OpenAI platform.openai.com 官方 API 是否支持 OAuth 账户用量查询？ | **做不到**。platform usage API 只对 `sk-` API Key 有效，OAuth 账户完全隔离。 |
| 2. 用 `auth.json.access_token` 能否查额度？ | **能做，且方案可靠**。Codex CLI 官方自己就是这么做的。 |
| 3. 同类开源工具怎么做？ | **已验证路径**：至少 `codex-switcher` / `codex-auth` / `CodexAccountSwitcher` 三个项目都在用同一个 endpoint。 |

---

## 1. OpenAI 官方 API —— 做不到

platform.openai.com 的 `/v1/usage`、`/v1/dashboard/billing/*` 都要求 API Key 鉴权，返回的是美元消费而非对话额度。ChatGPT Plus/Pro 订阅计费与 API 计费完全独立，官方至今未提供针对 OAuth 订阅用户的公开 usage endpoint。这条路彻底堵死，无需再验证。

## 2. `access_token` 调 chatgpt.com 内部接口 —— 能做

**Endpoint（已从 Codex 官方 Rust 源码 `codex-rs/backend-client/src/client.rs` 确认）：**

```
GET https://chatgpt.com/backend-api/wham/usage
Authorization: Bearer {tokens.access_token}
ChatGPT-Account-Id: {tokens.account_id}
User-Agent: codex-cli
```

**返回结构（`codex-rs/protocol/src/protocol.rs` 的 `RateLimitSnapshot`）：**

```jsonc
{
  "limit_id": "codex",
  "limit_name": "...",
  "primary":   { "used_percent": 23.5, "window_minutes": 300,  "resets_at": 1713600000 }, // 5 小时窗
  "secondary": { "used_percent": 41.2, "window_minutes": 10080,"resets_at": 1714032000 }, // 7 天窗
  "credits":   { "has_credits": false, "unlimited": false, "balance": null },
  "plan_type": "plus" | "pro" | "free" | "team" | "enterprise",
  "rate_limit_reached_type": null
}
```

**可直接渲染为 UI 的字段：`used_percent`（剩余 = 100 - used_percent）、`resets_at`（UNIX 秒，重置时间）、`plan_type`。** 这正是 Codex CLI `/status` 命令展示的数据源。

**可靠性评估：**

| 维度 | 结论 |
|---|---|
| 反爬 / Cloudflare | **低风险**。Codex CLI 的 TUI 每 60 秒轮询同一 endpoint（官方 issue #10869 为证），OpenAI 官方客户端自身在批量调用，不存在针对它的 bot challenge。 |
| Token 类型 | 用 `tokens.access_token`（不是 id_token）。本地 `~/.codex/auth.json` 已验证：access_token 的 JWT payload 带 `scp` scope 和 `https://api.openai.com/auth` claim，id_token 只是身份凭证无权限。 |
| Token 过期 | access_token 通常 1 小时过期，需用 `refresh_token` 调 `https://auth.openai.com/oauth/token` 续签。Codex CLI 在每次请求前自动 refresh，我们必须复刻这套逻辑，否则第 2 小时所有账户查询全部 401。 |
| 非官方风险 | endpoint 无公开文档，OpenAI 随时可能改路径、字段或加限流。但由于 Codex CLI 当前强依赖它，**至少在 Codex CLI 不大改的周期内（季度级别）稳定**。 |
| ToS 风险 | 理论上存在（`codex-auth` README 也发了警告），但调用方是用户自己的 token 查自己的用量，行为等价于 Codex CLI 本身。迄今未见封号案例。 |

## 3. 同类开源工具 —— 已趟过路

- **`loongphy/codex-auth`**（CLI）：README 明确列出调用 `/backend-api/wham/usage` + `/backend-api/accounts/check/v4-2023-04-27`（后者拿 team 名）。
- **`Lampese/codex-switcher`**（Tauri 桌面应用）：README 宣传"5-hour + weekly 实时用量"，字段与 `wham/usage` 的 primary/secondary 对齐。
- **`aikilan/CodexAccountSwitcher`**（macOS menu bar）：以 remaining % 展示，与 `used_percent` 字段一致。

三个独立实现互相印证，路径成熟。

## 产品建议

**V1.5.0 直接做额度展示，不需要降级方案。** 理由：

1. 技术路径已被官方客户端和三个开源项目验证，字段、鉴权、刷新逻辑全部可直接对齐 Codex CLI 源码。
2. 这是多账户切换器的**核心价值锚点**——没有用量对比，"切哪个账户"的决策就没有依据，产品会退化成书签管理器。
3. 实现成本可控：一个带 token refresh 的 HTTP client + 一个 `RateLimitSnapshot` 解析器，Node.js 侧 < 200 行代码。

**实现时必须内建的两道防线：**

- **Token refresh**：请求前检查 access_token `exp`，过期前 5 分钟主动用 refresh_token 续签并回写 `auth.json`，否则会破坏 Codex CLI 自身的登录状态。
- **接口失效降级**：若 `wham/usage` 返回 404/非预期结构，UI 自动降级为"手动标记最近触顶时间"（一个按钮 + 本地时间戳），并在设置里提示"OpenAI 接口变更，等待新版本"。这比直接白屏好。

**不要做**：缓存预测（基于历史消息数反推剩余额度）。OpenAI 的 `used_percent` 算法内部对不同模型权重不同（如 GPT-5.2 Thinking 比 GPT-5.2 贵得多），客户端复刻会严重偏差，不如直接查。

---

## 判断不了、需要验证的事

- **并发轮询多账户时是否触发限流**：Codex CLI 是单账户 60 秒一次，若我们同时查 10 个账户每 60 秒一次，是否会被 rate-limit？需要实测。建议首版错峰轮询（每账户 5 分钟间隔）+ 手动刷新按钮。
- **`plan_type` 的完整枚举值**：源码见到 `plus/pro/free/team/enterprise`，近期 issue #17419 提到"unknown WHAM plan values"（如 prolite），UI 要做未知值兜底。
