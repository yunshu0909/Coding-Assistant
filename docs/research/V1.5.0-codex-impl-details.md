# V1.5.0 Codex 实现细节补充调研

**调研时间**：2026-04-19
**目标**：为 refresh / 写文件 / 接口稳定性 / 并发限流 四件事给出生产级结论。前置阅读 `V1.5.0-codex-quota-research.md`。

---

## 1. refresh_token 续签请求规格

**结论**：按官方 Codex Rust 源码实现，是 **JSON body** 而非 form-urlencoded。

**endpoint**：`POST https://auth.openai.com/oauth/token`
**headers**：`Content-Type: application/json`（建议附 `User-Agent: codex_cli_rs/1.0.0 (...)`，非强制）
**body**：
```json
{ "client_id": "app_EMoamEEZ73f0CkXaXp7hrann",
  "grant_type": "refresh_token",
  "refresh_token": "<旧 refresh_token>" }
```
**响应**：`{ id_token, access_token, refresh_token, expires_in }`，所有字段都可能缺失——官方实现把 `id_token` / `refresh_token` 声明为 `Option`，缺省时**沿用旧值**。

**关键行为（直接决定实现）**：

- **refresh_token 是单次使用（Refresh Token Rotation）**：每次刷新返回一个新 refresh_token，旧的立即作废。若同一 refresh_token 被用两次，第二次返回 `401 { "error": "refresh_token_reused" }`（Codex issue #10332、#9634、#6036 都是这个坑）。
- **错误码分三类**（官方源码常量）：`refresh_token_expired` / `refresh_token_reused` / `refresh_token_invalidated`——全部属于永久失效，UI 必须提示"请重新登录"，不能自动重试。其余 5xx / 网络错误是瞬时，可退避重试。
- **必须原子写回 `~/.codex/auth.json`**：否则下次 Codex CLI 自己跑就用旧 refresh_token，立刻被 reused 而全盘崩。并且**要和 Codex CLI 自己的 refresh 抢锁**——这是下一节的核心风险。

**证据**：`codex-rs/login/src/auth/manager.rs`（官方，JSON body）、`Lampese/codex-switcher/src-tauri/src/auth/token_refresh.rs`（开源，form-urlencoded 也能通）、`openai/codex` issue [#10332](https://github.com/openai/codex/issues/10332)（确认 rotation）。

## 2. Codex 写 auth.json 的方式

**结论**：**官方不做原子写**，是 `truncate + write + flush`，且**没有跨进程锁**。

**官方实现**（`codex-rs/login/src/auth/storage.rs::FileAuthStorage::save`）：
```rust
options.truncate(true).write(true).create(true).mode(0o600);
file.write_all(json_data.as_bytes())?;
file.flush()?;
```
没有 `rename from tempfile`，意味着：
- **瞬时窗口内文件可能是半写状态**——chokidar 监听必须 debounce（建议 300–500ms），且解析失败要静默重试一次而不是报错。
- **Codex CLI 在 token 过期（默认 58 分钟）或 401 retry 时都会写一次**——不只是登录那一刻。
- **多实例并发写没保护**：issue #10332 已报告。我们的 Electron 进程如果和 Codex CLI 同时 refresh 同一账户，**100% 会触发 reused 错误**（两边都拿到新 refresh_token 但只有一个能落盘）。

**额外风险 —— 存储后端不一定是 auth.json**：官方支持 `AuthCredentialsStoreMode` = File / Keyring / Auto / Ephemeral。若用户 `cli_auth_credentials_store = "keyring"`，auth.json 根本不存在，credentials 在 macOS Keychain 里。**首版只支持 File 模式**，在设置里检测并提示用户切换，比盲目监听空文件更友好。

**对 chokidar 策略的结论**：
- debounce 300ms + 解析 JSON 失败静默重试；
- 以 `tokens.account_id` 为 diff 依据，不要用整体文件 hash（last_refresh 时间戳每次都变，会产生假阳性）；
- 主动写入前要 `usePolling: false` 下再做 `awaitWriteFinish` 兜底。

**证据**：`codex-rs/login/src/auth/storage.rs`（官方写文件逻辑，无 rename）、issue [#10332](https://github.com/openai/codex/issues/10332)（race 证明无跨进程锁）、[Codex CI/CD 文档](https://developers.openai.com/codex/auth/ci-cd-auth)（明确"不要在 CI 里重复 seed 原始 auth.json，会覆盖掉 Codex 已 refresh 的 token"）。

## 3. wham/usage 接口稳定性

**结论**：**endpoint 本身稳定，但 2026 年 4 月有近期 Cloudflare 层面 breakage，Linux 尤甚**。

- **endpoint 路径/字段本身**：2026.04 没有改路径或改返回结构的记录。`primary` / `secondary` / `plan_type` / `resets_at` 字段和 Codex Rust 源码 `RateLimitSnapshot` 结构体一致。三个开源项目（codex-auth、codex-switcher、CodexAccountSwitcher）近一月 commit 都没动 parser。
- **新增未知 plan_type 值**：issue 里出现过 `prolite`、`business`、`edu` 等新值（codex-auth 的 `parsePlanType` 已经列出 8 种）。**UI 必须做未知枚举兜底**（显示原字符串）。
- **Cloudflare 403 challenge**（真实风险，需要关注）：
  - issue [#17860](https://github.com/openai/codex/issues/17860)：Linux x86_64-musl 静态编译 rustls 被 Cloudflare 识别为 bot，**全部** chatgpt.com/backend-api/* 请求返回 403。macOS 的 native-tls 不受影响。
  - issue [#66633（openclaw）](https://github.com/openclaw/openclaw/issues/66633)：2026.04.14 某次升级后 UA 里 `originator: pi` 被 Cloudflare 标记为 bot，403 challenge 页。
  - **对我们的影响**：Electron 走系统 Node.js 的 `fetch()`（底层 undici + OS TLS），macOS 上不会触发；**Windows / Linux 需要真机实测**。UA 建议用 `codex_cli_rs/1.0.0` 这种已知白名单的前缀，别用 `codex-cli` 字面值。
- **ToS / 封号**：迄今 0 起公开案例，官方自己的客户端在同一 endpoint 每 60s 打一次。

**证据**：openai/codex issue [#10869](https://github.com/openai/codex/issues/10869)、[#17860](https://github.com/openai/codex/issues/17860)、[#17419](https://github.com/openai/codex/issues/17419)（prolite 未知 plan 值）。

## 4. 并发查多账户的限流

**结论**：**没有公开实测数据。三个开源项目的工程值可作参考：codex-auth 并发 3，codex-switcher 并发 10（上限写死）**。建议我们 V1 走 **并发 3 + 手动刷新 + 后台 5 分钟错峰**，不做自动轮询。

- codex-auth 文档明确："`list`/`switch` 批量刷新所有账户，**最大并发 3**"。这是对着实际用户账户数（通常 2–5 个）选的稳妥值。
- codex-switcher 源码 `refresh_all_usage` 用 `accounts.len().min(10).max(1)`——即 1–10 个账户同时查。没有降速，也没有报过限流 issue。
- Codex CLI 本体对**单账户 60s 一次**轮询，同时多实例并发（多个账户同时被不同进程查）在 issue #10869 的讨论里没有限流报告，只有"没必要打"的抱怨。
- **推测限流阈值**：OpenAI 的 chatgpt.com/backend-api 没有公开 rate-limit 文档；根据 ChatGPT 网页端经验（conversation endpoint 大约 60 rpm/user），同一 access_token **每分钟 10 次以内**应该绝对安全。多账户用不同 token 并行应当是独立计数，5 个账户同时查 ≈ 5 rpm。

**UI 策略建议**：
1. **初次打开页面**：`Promise.all` 批量查，但用 `p-limit(3)` 限流。5 个账户约 2–3 秒出结果。
2. **后台自动刷新**：默认关闭。若开启，**5 分钟错峰轮询**（每账户间隔 60s），而不是整点同时查。
3. **手动刷新**：用户点按钮，仍走并发 3。
4. **401 → refresh → retry one**（对齐 codex-switcher）；**403 连续 2 次 → 标红账户**提示"Cloudflare 疑似拦截"；**429 → 整表暂停 60s 并提示**。

**不确定的事（需要真机实测）**：
- Windows/Linux 上 Node.js fetch 是否触发 Cloudflare 403。
- 同一 access_token 的 rpm 硬阈值。
- 5 个账户同时打请求是否导致 account_id 级限流。
建议 V1 发布后埋点 `status_code` 分布，两周内可拿到真数据再调参数。

```js
// ========== 共用常量（来自官方 Codex Rust 源码） ==========
const OAUTH_CLIENT_ID = 'app_EMoamEEZ73f0CkXaXp7hrann'
const OAUTH_TOKEN_URL = 'https://auth.openai.com/oauth/token'
const USAGE_URL       = 'https://chatgpt.com/backend-api/wham/usage'
const USER_AGENT      = 'codex_cli_rs/1.0.0 (CodePal)' // 白名单前缀，降低 CF 误伤

/**
 * 刷新 access_token。成功后必须由调用方原子写回 auth.json。
 * @param {string} refreshToken 旧 refresh_token（单次使用）
 * @returns {Promise<{access_token, id_token?, refresh_token?, expires_in?}>}
 * @throws {CodexAuthPermanent} refresh_token 失效（expired/reused/invalidated），UI 提示重新登录
 * @throws {CodexAuthTransient} 5xx / 网络错误，可重试
 */
async function refreshAccessToken(refreshToken) {
  const res = await fetch(OAUTH_TOKEN_URL, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json', 'User-Agent': USER_AGENT },
    body: JSON.stringify({
      client_id: OAUTH_CLIENT_ID,
      grant_type: 'refresh_token',
      refresh_token: refreshToken,
    }),
  })
  if (res.ok) return res.json() // { access_token, id_token?, refresh_token?, expires_in? }

  const body = await res.text()
  if (res.status === 401) {
    // 永久失败：解析错误码给更准确的提示
    const code = tryParseErrorCode(body) // refresh_token_expired | _reused | _invalidated
    throw new CodexAuthPermanent(code || 'unknown', body)
  }
  throw new CodexAuthTransient(res.status, body) // 5xx / 其他
}

/**
 * 查询账户额度。内置 401→refresh→retry-once。
 * @param {{ access_token, refresh_token, account_id, id } } account 账户快照（会被就地更新 token）
 * @returns {Promise<RateLimitSnapshot>}
 * @throws {CodexAccountDead}  403 连续两次（账户被封或 Cloudflare 拦截）
 * @throws {CodexRateLimited}  429（UI 应全表冷却 60s）
 * @throws {CodexTransient}    5xx / 网络错误（UI 单卡片降级，30s 后重试一次）
 */
async function fetchRateLimit(account) {
  // 1. 过期预检：access_token JWT exp 在 60s 内过期，先主动 refresh
  if (isJwtNearExpiry(account.access_token, 60)) {
    await refreshAndPersist(account) // 原子写 auth.json + 更新内存
  }

  const send = (token) => fetch(USAGE_URL, {
    method: 'GET',
    headers: {
      'Authorization': `Bearer ${token}`,
      'ChatGPT-Account-Id': account.account_id,
      'User-Agent': USER_AGENT,
    },
  })

  let res = await send(account.access_token)

  // 2. 401 → refresh + retry once（对齐 codex-switcher）
  if (res.status === 401) {
    await refreshAndPersist(account)
    res = await send(account.access_token)
  }

  if (res.ok) return res.json()
  if (res.status === 401) throw new CodexAuthPermanent('token_invalid_after_refresh')
  if (res.status === 403) throw new CodexAccountDead(await res.text()) // 永久，建议删除账户
  if (res.status === 429) throw new CodexRateLimited()
  if (res.status >= 500) throw new CodexTransient(res.status) // 重试一次
  throw new CodexTransient(res.status, await res.text())       // 其他降级
}
```
